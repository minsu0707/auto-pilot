# v0.1.2-dev

## Status

- Current `develop` line
- Manifest version: `0.1.2-dev`
- Not a stable tag yet

## Focus

This line expands Auto Pilot from a basic intake-and-bootstrap flow into a more opinionated product-building workflow with stronger design direction, clearer runtime state, and a manager-led team model.

## Highlights

### UX and Entry Point

- Switched the primary public entry point to `$auto-pilot`
- Normalized multilingual docs and CLI output around the same public usage
- Aligned installer and README guidance around the newer interaction model

### Intake and Product Defaults

- Added `architecture_preset`
- Added `theme_preset`
- Added `visual_vibe`
- Kept `design_direction` as a refinement field instead of the main design decision
- Defaulted the recommended stack toward `Next.js + TypeScript + Tailwind CSS + shadcn/ui`

### Design Workflow

- Added `docs/design.md` for user-facing projects
- Added design synthesis guidance so UI work starts from a written design brief
- Added a visual workflow guide in `docs/05-how-it-works.md`
- Strengthened the rule that user-facing UI should go through explicit design review

### Runtime and State

- Expanded runtime state with execution mode, current owner, quality gates, planner checkpoint, reviewer verdict, and role results
- Expanded blocker records with `ownerRole`, `classification`, and `summary`
- Made `status` output team-aware
- Added `scripts/team_checkpoint.py` to persist planner, architect, builder, designer, QA, and manager checkpoints back into docs and state files

### Team Orchestration

- Introduced a manager-led orchestration model
- Added internal specialist skills:
  - `autopilot-planner`
  - `autopilot-architect`
  - `autopilot-designer`
  - `autopilot-qa`
- Added support for:
  - `team-product`
  - `team-lite`
  - `serial-fallback`

### Repository and Release Hygiene

- Added plugin metadata and policy files such as `LICENSE`, `SECURITY.md`, `PRIVACY.md`, and `TERMS.md`
- Added a repo-local plugin validation CI workflow
- Added version separation between stable and develop lines in the manifest
- Normalized README drift across languages and documented the stable install contract more clearly

## Before Stable Release

Before cutting the next stable tag from this line, make sure to update:

- stable install URLs in README files
- the installer's baked-in stable ref
- this folder with a new stable version note
